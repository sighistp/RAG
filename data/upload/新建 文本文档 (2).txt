# Requesting Code Review

Dispatch a code reviewer subagent to catch issues before they cascade. The reviewer gets precisely crafted context for evaluation — never your session's history. This keeps the reviewer focused on the work product, not your thought process, and preserves your own context for continued work.

**Core principle:** Review early, review often.

## When to Request Review

**Mandatory:**
- After each task in subagent-driven development
- After completing major feature
- Before merge to main

**Optional but valuable:**
- When stuck (fresh perspective)
- Before refactoring (baseline check)
- After fixing complex bug

## How to Request

**1. Get git SHAs:**
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # or origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

**2. Dispatch code reviewer subagent:**

Use Task tool with `general-purpose` type, fill template at `code-reviewer.md`

**Placeholders:**
- `{DESCRIPTION}` - Brief summary of what you built
- `{PLAN_OR_REQUIREMENTS}` - What it should do
- `{BASE_SHA}` - Starting commit
- `{HEAD_SHA}` - Ending commit

**3. Act on feedback:**
- Fix Critical issues immediately
- Fix Important issues before proceeding
- Note Minor issues for later
- Push back if reviewer is wrong (with reasoning)

## Integration with Workflows

**Subagent-Driven Development:**
- Review after EACH task
- Catch issues before they compound
- Fix before moving to next task

**Executing Plans:**

- Get feedback, apply, continue

**Ad-Hoc Development:**
- Review before merge
- Review when stuck

**Never:**
- Skip review because "it's simple"
- Ignore Critical issues
- Proceed with unfixed Important issues
- Argue with valid technical feedback

`asyncio.run()` creates a new event loop in a thread, but `httpx.AsyncClient` created inside `wm.notify()` may not be thread-safe across loops. ```
This hardcodes the assumption that `project_dir` is at `<parent>/projects/<name>`. If `settings.project_dir` is an absolute path (e.g., `/data/my-projects`), `.parent.parent / "projects"` produces a wrong path. The snapshot won't match the actual project location.

### I5: `trace_db.py:10` — Hardcoded relative DB path
```python

```
Relative to CWD, not to the module. If the server is started from a different directory, traces will be written to an unexpected location. Should use `Path(__file__).parent.parent / "data" / "traces.db"`.

### I6: `test_deliver_hooks.py` — Tests mutate global `settings.project_dir`
Lines 27-37 save/restore `settings.project_dir` in try/finally. This is not thread-safe and creates fragile test coupling. Use `monkeypatch.setattr` instead (which auto-restores).

### I7: `security_scanner.py:11` — Custom pattern too broad
```python

```
Matches any string assignment to a variable containing "secret", "password", etc. — including test fixtures, documentation strings, and example code. Will produce false positives on most real projects.

### I8: `.env` file exists on disk but not in `.gitignore`
Both `.devteam_secret` and `.BLUEPRINT_secret` are on disk alongside `.env`. If these aren't gitignored, they'll be committed.
